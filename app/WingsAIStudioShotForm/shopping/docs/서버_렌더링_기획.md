# 쇼핑 숏폼 영상 다운로드 — 서버 렌더링 기획

---

## 한 페이지 요약 (구현 시 참고)

| 구분 | 내용 |
|------|------|
| **목표** | 다운로드 시 클라이언트(Canvas+MediaRecorder) 대신 **서버(Cloud Run)**에서 영상 렌더링 → 모바일/Safari 끊김·메모리 이슈 해소 |
| **흐름** | [다운로드 클릭] → 클라이언트가 TTS/영상3개/썸네일을 **URL 또는 base64로 준비** → **POST /api/ai/render** (body에 `type: "shopping"`) → Next.js가 Cloud Run으로 전달 → Cloud Run이 **썸네일+영상3개 concat+자막+오디오** 합성 → **videoUrl** 또는 **videoBase64** 반환 → 클라이언트가 다운로드 |
| **클라이언트** | ① blob URL이면 Supabase 등에 업로드 후 공개 URL 취득 ② `handleRenderVideo()`를 **서버 요청**으로 교체 (payload 직렬화) ③ 응답으로 받은 URL/base64 → Blob → 다운로드 |
| **API** | 기존 `POST /api/ai/render` 유지. body에 `type: "shopping"` 넣으면 쇼핑 전용 필드(`videoSegments`, `thumbnailImage*`, `subtitles`, 오디오, BGM/효과음) 사용 |
| **Cloud Run** | `type === "shopping"`이면: 썸네일(0~0.01초) + `videoSegments[]` 3개 concat + 자막 + TTS+BGM+효과음 믹싱 → 1080×1920 MP4 등으로 출력 후 URL 또는 base64 반환 |
| **구현 순서** | ① 스펙 확정 ② blob→URL 업로드 유틸 ③ 클라이언트 다운로드→서버 요청 교체 ④ Next.js API에서 `type===shopping` 분기 ⑤ Cloud Run 쇼핑 파이프라인 구현 ⑥ (선택) 서버 실패 시 클라이언트 렌더 폴백 |

## 1. 목표

- **현재**: 클라이언트(브라우저)에서 Canvas + MediaRecorder로 최종 영상을 렌더링 후 다운로드  
  → 모바일에서 끊김, Safari WebM 미지원, 메모리/타임아웃 이슈
- **목표**: 롱폼처럼 **서버(Cloud Run 등)**에서 영상을 렌더링하고, 클라이언트는 요청·다운로드만 담당

---

## 2. 전체 데이터 흐름

```
[쇼핑 숏폼 화면]
       │
       │ ① "다운로드" 클릭
       ▼
[클라이언트] blob URL → 업로드(Supabase 등) → 공개 URL 확보
       │
       │ ② POST /api/ai/render  (body: type: "shopping", ...)
       ▼
[Next.js API] type 분기 → 쇼핑 payload 그대로 Cloud Run으로 전달
       │
       │ ③ POST {CLOUD_RUN_URL}/render
       ▼
[Cloud Run] 쇼핑 전용 파이프라인: 썸네일 + 영상3개 concat + 자막 + 오디오 믹싱 → MP4
       │
       │ ④ { videoUrl } 또는 { videoBase64 }
       ▼
[클라이언트] URL이면 fetch → Blob → 다운로드 / base64면 디코딩 → Blob → 다운로드
```

## 3. 롱폼 vs 쇼핑 — 서버 입력 비교

| 구분 | 롱폼 (기존) | 쇼핑 (신규) |
|------|-------------|-------------|
| 구분 키 | `type` 없음 또는 `"longform"` | `type: "shopping"` |
| 비디오 소스 | `characterImage`(1장) + `autoImages[]`(구간별 이미지) | `thumbnailImage*`(0.01초) + `videoSegments[]`(영상 3개 URL) |
| 오디오 | `audioBase64` / `audioGcsUrl` | 동일 |
| 자막 | `subtitles[]` (id, start, end, text) | 동일 형식 |
| BGM/효과음 | (롱폼은 별도 처리 여부 확인) | `bgmUrl`, `bgmStartTime`, `bgmEndTime`, `sfxUrl`, `sfxStartTime`, `sfxEndTime` |
| 해상도 | config.width/height (예: 1920×1080) | 1080×1920 (9:16) 고정 |
| duration | 전체 길이(초) | 동일 |

## 4. 롱폼 서버 렌더링 구조 (참고)

| 항목 | 내용 |
|------|------|
| API | `POST /api/ai/render` (Next.js) |
| 실제 연산 | **Cloud Run** 서비스 (`CLOUD_RUN_RENDER_URL` → `/render`) |
| 입력 | `audioBase64` 또는 `audioGcsUrl`, `subtitles[]`, `characterImage`/URL, `autoImages[]`, `duration`, `config` |
| 출력 | `videoUrl` 또는 `videoBase64` |
| 긴 영상 | 15분 이상 시 클라이언트가 Cloud Run에 직접 요청 가능 (`NEXT_PUBLIC_CLOUD_RUN_RENDER_URL`) |

## 5. 쇼핑 숏폼 렌더 스펙 (서버 입력)

클라이언트가 “다운로드” 시 서버/렌더러에 넘길 payload 정의.

### 3.1 공통

- **해상도**: 1080×1920 (9:16)
- **fps**: 30
- **타임라인**: 썸네일(0~0.01초) → 영상 1 → 영상 2 → 영상 3, TTS/자막/BGM·효과음 동기화

### 5.2 서버에 넘길 데이터

| 필드 | 타입 | 설명 |
|------|------|------|
| `type` | `"shopping"` | 롱폼과 구분용 (같은 엔드포인트 쓸 경우) |
| `duration` | number | 전체 길이(초). TTS 길이 기준 |
| `audioBase64` | string (선택) | TTS 오디오 base64. 4.5MB 초과 시 `audioGcsUrl` 사용 권장 |
| `audioGcsUrl` | string (선택) | TTS 오디오 GCS URL (Vercel body 크기 제한 회피) |
| `subtitles` | `{ id, start, end, text }[]` | 자막. 의미 단위(이미 클라이언트에서 계산된 scriptLines) |
| `thumbnailImage` / `thumbnailImageUrl` / `thumbnailImageGcsUrl` | string | 맨 앞 0.01초에 보여줄 썸네일 (base64 또는 URL) |
| `videoSegments` | `{ url: string, startTime: number, endTime: number }[]` | 영상 3개. 각각 재생 구간(초). 순서대로 이어붙임 |
| `bgmUrl` (선택) | string | BGM 오디오 URL |
| `bgmStartTime` / `bgmEndTime` | number | BGM 재생 구간(초) |
| `sfxUrl` (선택) | string | 효과음 URL |
| `sfxStartTime` / `sfxEndTime` | number | 효과음 재생 구간(초) |
| `config` | `{ width: 1080, height: 1920, fps: 30 }` | 쇼핑은 1080×1920(세로) 고정 |

- 오디오: `audioBase64` 또는 `audioGcsUrl` 중 하나 필수.  
- 썸네일: base64 크기 제한 걸리면 URL/GCS 전달 방식 추가.
- `videoSegments`의 `url`은 공개 접근 가능한 URL이어야 함 (blob은 서버에서 접근 불가 → 업로드 후 URL 전달).

## 6. 구현 경로 (선택)

### 옵션 A: 기존 Cloud Run 롱폼 렌더러 확장

- **장점**: 인프라 하나로 통일, 인증/배포 일원화  
- **단점**: 롱폼 로직과 쇼핑 로직이 달라 서비스 내부 분기 필요  
- **요구**: Cloud Run 서비스에서 `type === "shopping"`이면  
  - 썸네일 이미지 0.01초 표시  
  - `videoSegments` 3개를 순서대로 concat  
  - 자막·TTS·BGM/효과음 믹싱  
  (롱폼은 characterImage + autoImages 구조이므로 쇼핑 전용 파이프라인 필요)

### 옵션 B: 쇼핑 전용 API + 전용 렌더 엔드포인트

- **예**: Next.js `POST /api/shopping/render` → Cloud Run `POST /render/shopping` (또는 기존 URL에 `type: "shopping"` body)  
- **장점**: 쇼핑 스펙만 명확히 정의, 롱폼 코드와 분리  
- **단점**: Cloud Run에 쇼핑용 처리 추가 필요

### 옵션 C: 동일 API 라우트, body로 타입 구분 (권장)

- **API**: 기존 `POST /api/ai/render` 유지  
- **body**:  
  - `type` 없거나 `"longform"` → 기존 롱폼 로직  
  - `type: "shopping"` → 쇼핑 전용 필드만 사용 (`videoSegments`, `thumbnailImage*`, `subtitles`, 오디오 등)  
- **Cloud Run**: `/render`에서 `type` 읽어 쇼핑/롱폼 분기  
- **장점**: 클라이언트는 URL 하나만 알면 됨, 환경 변수 공유

## 7. 클라이언트(쇼핑) 변경 사항

1. **다운로드 버튼 동작**
   - 기존: `handleRenderVideo()` → Canvas + MediaRecorder로 클라이언트 렌더링 후 Blob 다운로드  
   - 변경:  
     - 미리보기 생성은 그대로 두고,  
     - “다운로드” 클릭 시 **서버 렌더 요청**만 수행  
     - 요청 body: 위 3.2 스펙으로 직렬화 (scriptLines → subtitles, convertedVideoUrls → videoSegments 등)

2. **자료 전달 시 제약**
   - **TTS 오디오**: blob이면 먼저 Supabase 등에 업로드 → `audioGcsUrl` 또는 공개 URL로 전달.  
     또는 base64로 보내되 4.5MB 이하일 때만 (Vercel 제한).
   - **영상 3개**: `convertedVideoUrls`가 blob URL이면 서버에서 접근 불가.  
     → 업로드 후 공개 URL 배열로 `videoSegments[].url` 전달.
   - **썸네일**: base64 또는 이미 업로드된 URL.

3. **진행/에러 처리**
   - “다운로드” 클릭 시 로딩 UI (예: “서버에서 영상 생성 중…”)
   - `POST /api/ai/render` (또는 쇼핑 전용 API) 호출 → `videoUrl` 또는 `videoBase64` 수신
   - `videoUrl`이면 `fetch`로 Blob 받아서 다운로드 트리거; `videoBase64`면 디코딩 후 Blob으로 저장

4. **폴백**
   - 서버/Cloud Run 실패 시: 기존처럼 클라이언트 렌더링 시도 (선택 사항)

## 8. 서버(Cloud Run) 쇼핑 렌더 로직 (개요)

1. **입력 검증**  
   `type === "shopping"`이고, `videoSegments` 3개, 오디오, `subtitles`, 썸네일 존재 확인.

2. **타임라인 생성**
   - 0 ~ 0.01초: 썸네일 이미지
   - 이후: `videoSegments[0].url`, `[1].url`, `[2].url`을 각각 `startTime`~`endTime` 구간만큼 concat (또는 구간 정보로 잘라서 이어붙임)

3. **오디오**
   - TTS + (선택) BGM 구간 믹싱 + (선택) 효과음 구간 믹싱  
   - FFmpeg 등으로 단일 오디오 트랙 생성

4. **자막**
   - `subtitles[]`를 SRT/ASS 또는 FFmpeg drawtext 필터 입력으로 변환  
   - 의미 단위·줄바꿈은 이미 클라이언트에서 나온 값 그대로 사용

5. **합성**
   - 비디오 트랙(썸네일+3개 세그먼트) + 오디오 트랙 + 자막 → 최종 MP4/WebM  
   - 1080×1920, 30fps 유지

6. **반환**
   - GCS 업로드 후 `videoUrl` 반환 또는 바이너리 base64 반환 (기존 롱폼과 동일 정책)

## 11. 정리

- **왜 서버 렌더링인가**: 모바일/Safari에서의 끊김·코덱 제한·메모리 제한을 피하고, 롱폼과 동일한 방식으로 안정적인 다운로드를 제공하기 위함.
- **핵심**:  
  - 클라이언트는 “미리보기”만 담당하고, “다운로드”는 서버에 넘길 payload만 준비.  
  - 모든 미디어(오디오, 영상 3개, 썸네일)는 서버가 접근 가능한 URL 또는 base64로 전달.  
  - Cloud Run(또는 동일 인프라)에 쇼핑 전용 타임라인·자막·오디오 믹싱 로직을 추가하면 됨.

이 기획대로 진행하면 “영상 다운로드는 롱폼처럼 서버에서 렌더링”이 가능해진다.
