# 정교한 대본이 끝까지 나오는 원리 및 구현 가이드

다른 프로그램에서 동일한 방식을 적용할 때 참고할 수 있도록, **정교한 대본이 중간에 끊기지 않고 끝까지 생성되는 원리**와 **구현 시 지시사항**을 정리한 문서입니다.

---

## 1. 문제 정의

- LLM으로 긴 대본(수천~수만 자)을 한 번에 생성하면:
  - **출력 토큰 한도(max_tokens)** 에 걸려 중간에 잘리거나
  - **응답이 문장 중간에서 끊겨** 부자연스러운 결과가 나올 수 있음.
- 목표: **항상 “문장이 완결된 상태”로 끝나도록** 하고, 필요한 분량은 **끊김 없이 전부** 나오게 하는 것.

---

## 2. 해결 원리 (두 가지 축)

| 축 | 내용 | 목적 |
|----|------|------|
| **분할 생성** | 긴 대본을 시간/구간별로 잘라, **한 번에 생성하는 분량을 작게** 만듦 | 한 회차당 출력이 토큰 한도 내에서 끝나기 쉽게 함 |
| **끊김 감지 + 후처리/이어쓰기** | 생성 결과가 문장 중간에서 끊겼는지 검사하고, 끊겼으면 **이어쓰기 호출** 또는 **마지막 완전한 문장까지만 사용** | 중간에 끊긴 문장으로 끝나지 않게 함 |

---

## 3. 구현 시 할 일 (명령 사항)

다른 프로그램에서 “정교한 대본처럼 끝까지 나오게” 하려면 아래를 순서대로 적용하면 됩니다.

---

### 3.1 분할 규칙 (언제 몇 회로 나눌지)

- **대본 길이(재생 시간)** 에 따라 **전체를 몇 회차로 나눌지** 결정합니다.

| 대본 길이(분) | 분할 수(totalParts) | 비고 |
|---------------|---------------------|------|
| 5분 | 1 | 한 번에 생성 |
| 10분 | 2 | 0~4:30, 4:30~10:00 |
| 15분 | 3 | 0~4:30, 4:30~9:00, 9:00~15:00 |
| 20분 | 4 | 동일 패턴으로 4등분 |
| 25분 | 5 | 5등분 |
| 30분 | 6 | 6등분 |
| 35분 | 7 | 7등분 |
| 40분 | 8 | 8등분 |

- **한 회차당 목표 글자수**: 대략 **2,070자 ~ 2,400자** 구간으로 두면, 한 번의 API 호출로 끝까지 나올 가능성이 높음.  
  - 예: 전체 목표가 6,000자면 3회차로 나누어 회차당 약 2,000~2,100자.

---

### 3.2 회차별 API 호출 방식

- **5분(1회차만)**  
  - 한 번의 API 호출로 전체 생성.  
  - 아래 3.4의 **출력 토큰 한도**와 3.5의 **MAX_TOKENS 대응**을 반드시 적용.

- **10분 이상(2회차 이상)**  
  - `partNumber = 1, 2, ..., totalParts` 순서로 **순차 호출**.  
  - 각 호출에는 다음을 넘깁니다.  
    - **기획안(또는 전체 지시문)**  
    - **이번 회차의 시간 구간** (예: "0분~4분 30초", "4분 30초~10분")  
    - **이번 회차 목표 글자수** (예: 2,150자)  
  - 2회차부터는 **이전 회차까지 생성된 텍스트의 마지막 N자(예: 500자)** 를 컨텍스트로 넘겨 이어서 쓰게 함.  
  - 모든 회차가 끝나면 **순서대로 이어붙여** 최종 대본으로 사용.

---

### 3.3 끊김 감지 (한 번에 받은 텍스트가 끝까지 완결됐는지)

생성된 문자열 `content`에 대해 아래를 검사합니다.

1. **마지막 문장 끝 위치**  
   - `.` `?` `!` `。` `？` `！` 중 **가장 뒤에 나오는 위치**를 `lastSentenceEnd`로 둡니다.

2. **끊긴 것으로 판단하는 조건 (하나라도 만족하면 “끊김”)**  
   - **마지막 50자 안에** 위 문장 끝 부호가 **없음**.  
   - **마지막 100자**를 잘라서 봤을 때, **끝이 문장 부호로 끝나지 않음** (`endsWithPunctuation == false`).  
   - **마지막 단어**가 불완전한 어미로 끝남 (예: `~하`, `~해`, `~을`, `~를`, `~이`, `~에`, `~와`, `~과`, `~로`, `~의` 등 한글 조사/어미만 있는 경우).  
   - **마지막 2단어**가 “문제 해”, “재조명 하”, “해결 하”, “설명 하”, “이야기 하” 등 **동사가 잘린 구문**으로 끝남.

3. **반환값**  
   - `isTruncated`: 위 조건으로 본 “끊김 여부” (true/false).  
   - `lastCompleteSentenceEnd`: **마지막으로 완전한 문장이 끝난 위치** (문장 부호 다음 인덱스).  
     - 문장 부호가 없으면 -1 또는 0으로 두고, “완전한 문장 끝”이 없다고 처리.

**구현 시 지시:**  
- “끊김 감지” 함수를 하나 두고, **모든 ‘한 번에 생성된’ 텍스트**에 대해 1~3을 적용해 `isTruncated`와 `lastCompleteSentenceEnd`를 계산하도록 합니다.

---

### 3.4 출력 토큰 한도 (한 번에 생성할 때)

- **한 번에** 긴 대본을 생성하는 경우(예: 5분 분량 한 번에), **출력이 토큰 한도에 걸리지 않도록** `max_tokens`(또는 `maxOutputTokens`)를 넉넉히 줍니다.

권장 계산식:

- `maxOutputTokens = min(65536, max(16384, ceil(목표글자수 * 5)))`  
  - 한글은 대략 1자당 2~3토큰 가정하고, 여유 있게 5배를 준 뒤, 상한 65536, 하한 16384로 클램프.

**구현 시 지시:**  
- “정교한 대본”처럼 **한 번에 5분 분량**을 만들 때는 반드시 위 식으로 `max_tokens`를 설정하고, **한 회차당 2,000자대**로 나눌 때는 회차당 2,000*5 수준 이상은 주도록 합니다.

---

### 3.5 API 응답의 “끝까지 안 나온 경우” 처리 (MAX_TOKENS·finishReason)

- API 응답에 **finishReason** (또는 equivalent)이 있으면 반드시 확인합니다.

- **finishReason === "MAX_TOKENS"**  
  - “출력이 토큰 한도 때문에 잘렸다”는 의미.  
  - **재시도** 시:  
    - **입력 토큰을 줄인 뒤** 같은 대본을 다시 생성 (기획안을 50%만 쓰거나, 앞 40% + 뒤 60%만 유지하는 식으로 축약).  
    - 출력 토큰 한도는 그대로 두고, **입력만 축약**하면 같은 한도로도 끝까지 나올 가능성이 커짐.

- **finishReason === "SAFETY" / "RECITATION" / "OTHER"**  
  - 재시도해도 동일할 가능성이 크므로, **재시도하지 말고** 사용자에게 안내.

**구현 시 지시:**  
- MAX_TOKENS일 때만 **기획안(또는 프롬프트) 축약 → 같은 요청으로 재호출** 로직을 넣고, 최대 재시도 횟수(예: 2~3회)를 정해 두세요.

---

### 3.6 끊긴 문장 “이어쓰기” (선택이지만 권장)

- 끊김 감지에서 **isTruncated == true** 이고, **lastCompleteSentenceEnd > 0** 이면:

  1. **완성된 부분**: `content.substring(0, lastCompleteSentenceEnd)`  
  2. **끊긴 부분**: 그 뒤의 문자열 (문장이 중간에 끊긴 부분)

- **이어쓰기용 API 호출** 시 다음을 넘깁니다.  
  - 주제/기획안  
  - “이미 작성된 부분” (완성된 부분의 마지막 200~500자)  
  - “끊어진 문장의 시작” (끊긴 부분의 앞 50자 정도)  
  - 지시문: “위 내용을 이어서 끊어진 문장을 완성하고, 반드시 완전한 문장(마침표/물음표/느낌표)으로 끝내라.”

- 응답을 받으면 **완성된 부분 + 이어쓰기 결과**를 합쳐서 한 덩어리로 저장하고, **다시 끊김 감지**를 수행.  
  - 여전히 끊겼으면 **최대 N회(예: 2회)** 까지 이어쓰기를 반복.

**구현 시 지시:**  
- “한 회차” 또는 “한 번에 생성” 결과에 대해 끊김 감지 후, true이면 **이어쓰기 1회** 이상 수행하고, 그 결과를 해당 회차의 최종 텍스트로 사용하도록 합니다.

---

### 3.7 최종 후처리 (마지막 문장 정리 + 메타데이터 제거)

- **마지막 문장이 불완전한 경우**  
  - 끊김 감지와 동일한 방식으로 “마지막 완전한 문장 끝” 위치를 구한 뒤,  
    **그 위치까지만 잘라서 사용** (`content = content.substring(0, lastCompleteSentenceEnd)`).  
  - 문장 부호가 하나도 없으면, **맨 끝에 `.` 하나**만 붙여서 완결 처리.

- **메타데이터 제거**  
  - 대본 본문이 아닌 문구는 제거합니다.  
    - 예: `(공백 포함 2314자)`, `(글자수: 2297자)`, `(다음 회차로 이어집니다.)`, `(다음 편으로 이어집니다.)`, `(계속됩니다.)`, `(이어집니다.)` 등.  
  - 정규식 또는 문자열 치환으로 제거한 뒤 `trim()` 적용.

**구현 시 지시:**  
- 모든 회차를 이어붙인 **최종 대본**에 대해  
  1) 메타데이터 제거,  
  2) 마지막 문장 완결 처리(잘림 시 마지막 완전한 문장까지만 자르기)  
  를 적용한 결과를 사용자에게 보여주도록 합니다.

---

## 4. 흐름 요약 (다른 프로그램에 넣을 때)

1. **대본 길이(분) 입력** → 분할 수 `totalParts`와 회차당 목표 글자수 결정.
2. **totalParts === 1** (5분 등)  
   - 한 번에 생성.  
   - `maxOutputTokens`는 3.4 식 적용.  
   - 응답의 `finishReason` 확인 → MAX_TOKENS면 기획안 축약 후 재시도(3.5).
3. **totalParts >= 2**  
   - 1회차부터 순차 호출.  
   - 각 회차는 “해당 시간 구간 + 목표 글자수”만 넣어 회차당 2,000자대 생성.  
   - 2회차부터는 이전 회차 텍스트 마지막 N자를 컨텍스트로 포함.
4. **각 회차(또는 한 번에 생성된 덩어리)에 대해**  
   - 끊김 감지(3.3) → true이면 이어쓰기(3.6) 또는 “마지막 완전한 문장까지만 사용”.  
   - 메타데이터 제거(3.7).
5. **전체를 이어붙인 뒤**  
   - 다시 한 번 “마지막 문장 완결” 검사 및 자르기(3.7).  
   - 이 결과를 “끝까지 나온 정교한 대본”으로 사용.

---

## 5. 정리

- **끝까지 나오게 하는 핵심**  
  - **분할**: 한 번에 생성하는 분량을 2,000자대(또는 5분) 수준으로 제한.  
  - **토큰 한도**: 한 번에 생성할 때는 `max_tokens`를 목표 글자수의 5배 이상 등으로 넉넉히.  
  - **MAX_TOKENS 시**: 입력(기획안)만 줄여서 재시도.  
  - **끊김 감지**: 문장 끝 부호·불완전한 어미로 “끊긴 출력” 판별.  
  - **이어쓰기 또는 자르기**: 끊겼으면 이어쓰기 호출 또는 “마지막 완전한 문장”까지만 사용.  
  - **후처리**: 메타데이터 제거 + 최종 문장 완결 처리.

위 순서와 규칙만 지키면 다른 프로그램에서도 “정교한 대본이 끝까지 나오는” 것과 동일한 원리로 구현할 수 있습니다.
