# 렌더링 속도 분석: 2 CPU vs 4 CPU

## CPU와 렌더링 속도의 관계

### 렌더링 작업의 특성
- **CPU 집약적**: FFmpeg를 사용한 영상 인코딩
- **멀티코어 활용**: FFmpeg는 멀티스레드 처리 가능
- **병렬 처리**: 이미지 처리, 오디오 처리, 인코딩을 병렬로 수행

### CPU 증가의 효과

#### 현재 (2 CPU)
```
FFmpeg 작업:
- 2개 코어로 병렬 처리
- CPU 사용률: 100% (항상 최대 사용)
- 병목 현상: CPU가 부족하여 대기 시간 발생
```

#### 업그레이드 후 (4 CPU)
```
FFmpeg 작업:
- 4개 코어로 병렬 처리
- CPU 사용률: 100% (더 많은 작업 병렬 처리)
- 병목 현상 감소: CPU 여유로 대기 시간 감소
```

**개선율: 약 50% 빠름** ✅

## 실제 개선율 분석

### 이론적 개선
- CPU 2배 증가 → 속도 2배 향상 (이상적)
- 실제: 약 1.5-2배 향상 (다른 요인 고려)

### 실제 개선율 (보수적 추정)

**FFmpeg 인코딩:**
- CPU 집약적 작업: 약 1.8-2배 빠름
- 멀티스레드 활용: 4코어로 효율적

**이미지 처리:**
- 병렬 처리 가능: 약 1.5-1.8배 빠름

**오디오 처리:**
- CPU 의존적: 약 1.5-2배 빠름

**전체 렌더링:**
- **평균 개선율: 약 1.5-2배 빠름** ✅
- **실제 소요 시간: 약 50% 단축** ✅

## 추가 개선 요소

### 메모리 증가 (4GB → 8GB)의 효과

**현재 (4GB):**
- 메모리 부족 시 스왑 사용
- 디스크 I/O 증가 → 속도 저하

**업그레이드 후 (8GB):**
- 메모리 여유로 스왑 사용 감소
- 디스크 I/O 감소 → 속도 향상
- **추가 개선: 약 5-10%** ✅

### Concurrency 1의 효과

**현재 (Concurrency 80):**
- 여러 요청이 같은 인스턴스에서 경쟁
- 리소스 경쟁으로 속도 저하

**업그레이드 후 (Concurrency 1):**
- 각 요청이 전체 리소스 독점 사용
- 리소스 경쟁 없음 → 속도 향상
- **추가 개선: 약 10-20%** ✅
